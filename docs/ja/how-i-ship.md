## この文書が説明すること

この文書は、**一つのソフトウェア変更を、情報収集から本番反映の手前（マージ判断）まで、どう進めるか**を説明します。

ここで扱うのは「AI ツールのカタログ」ではありません。次の問いに答えるための文書です。

| 問い | 答えの置き場 |
|------|----------------|
| 何から始めるか | **情報収集と壁打ち**（いきなりコードを書かない） |
| 作業の単位は何か | **GitHub Issue 1 本**（受入条件つき） |
| 誰がコードを書くか | 主に **Claude Code**（ブランチ上） |
| 誰が厳しく見るか | **Codex**（Issue に照らした敵対レビュー） |
| 誰が決めるか | **自分**（指摘の採否・マージ） |

**関連する別文書**

- もっと前段の「観察→判断→Issue になるまで」は [知識が積み上がる仕組み](./how-knowledge.md)
- Claude に任せる／任せない境界だけは [ここでの Claude Code](./how-claude.md)

## 出発点は情報収集

変更の話をする前に、**いまのリポジトリと状況について何が分かっているか**を集めます。ここを飛ばすと、壁打ちも実装も空中戦になります。

### 何を集めるか

| 種類 | 例 |
|------|-----|
| リポジトリ上の事実 | ディレクトリ構成、既存の実装、テスト、設定、関連 docs / ADR |
| 変更のきっかけ | バグ報告、要望、自分のメモ、障害の記録 |
| 制約 | 期限、触ってはいけない範囲、セキュリティ・PII |
| 受入の材料 | 「誰が・どの画面で・何ができれば成功か」 |

### ChatGPT による壁打ち（リポジトリ接続）

壁打ちに使う ChatGPT は、**実際のリポジトリに接続された状態**で使います。

そのため:

- 抽象的な一般論ではなく、**そのリポジトリのコードと構造を踏まえた**検討ができる  
- 「どこを触るべきか」「似た実装が既にないか」を、手元の推測だけに頼らず話せる  
- 確度の高い壁打ちになる（ただしこの段階ではまだ **Issue にはしない**）

壁打ちで出したいものは次のようなものです。

- 問題の言い直し（誰の何が困っているか）
- 取りうる方針の候補と、それぞれのコスト・リスク
- 実装に入る前に人間が決めるべき論点
- Issue に書くべき受入条件の下書き

まだ「確定した仕様」ではありません。**選択肢と論点**を揃える段階です。

## 誰が何をするか（全体）

| 責任 | 担当 | 具体的にすること |
|------|------|------------------|
| 情報収集・壁打ち | 自分 + ChatGPT（リポジトリ接続） | 事実と制約を集め、方針案を検討する |
| スコープを確定する | 自分 + GitHub Issue | 受入条件と「完了」の定義を Issue に書く |
| 実装する | Claude Code | ブランチ上でスコープ内の変更を行う |
| 厳しくレビューする | Codex | Issue に照らして差分を突く |
| 指摘を採否する | 自分 | 直す／捨てる／スコープを疑う |
| PR を開く | Claude Code | Issue リンクと経緯を載せた PR を作る |
| マージする | 自分 | main に入れるか、Issue に戻すか |

図の色は、おおむね次の対応です。

| 色の系統 | 役割 |
|----------|------|
| ティール系 | ChatGPT / Codex（考える・突く） |
| 藍系 | Claude Code（実装・PR） |
| 緑系 | 自分（スコープ・判断・マージ） |
| GitHub マーク | Issue / PR という制御の器 |

## 手順（図と同じ 1〜7）

### Plan — 何を変えるかを決める

#### 1. 変更をフレーミングする（ChatGPT）

**目的:** 問題と方針の候補を、リポジトリの事実に根ざして整理する。

**やること:**

1. きっかけ（バグ・要望・メモ）を短く書く  
2. リポジトリ接続の ChatGPT に、現状の実装・類似箇所・制約を踏まえて議論する  
3. 「やること／やらないこと」の仮案を出す  
4. まだ Issue を確定させない（情報が足りなければ収集に戻る）

**終わったと言える状態:**

- 問題を一文で言える  
- 方針が 1 つ以上ある  
- 人間が決めるべき論点が列挙されている  

#### 2. スコープと受入を定義する（自分 · GitHub Issue）

**目的:** 以降の実装とレビューの**唯一の基準**を Issue にする。

**Issue に書くもの（最低限）:**

| 項目 | 内容 |
|------|------|
| 何を変えるか | 対象画面・API・データなど |
| 何を変えないか | 今回スコープ外 |
| 受入条件 | 完了とみなす具体条件 |
| 根拠 | 関連 docs / ADR / 調査メモへのリンク |
| リスク | 壊しやすい箇所、PII、権限 |

**終わったと言える状態:**

- 別の人が読んでも「完了条件」が同じ理解になる  
- 実装者が Issue 以外を正にしなくてよい  

ここから先は、**Issue が契約**です。壁打ちログは参考資料です。

---

### Build · review — 作って、突いて、直す（ループ）

#### 3. 実装する（Claude Code）

**目的:** Issue の受入を満たす変更を、ブランチ上に作る。

**やること:**

- 専用ブランチで作業する（main 直 push はしない）  
- スコープ外の「ついでリファクタ」を入れない  
- 自動で回せる tests / build を通す  

**終わったと言える状態:**

- 受入条件に対応する差分がある  
- ブランチ上でビルドやテストが必要な範囲まで通っている  

手元での操作確認（hands-on）は、変更の性質によっては自分が行う。Claude だけに「全部確認した」としない。

#### 4. 敵対レビュー（Codex）

**目的:** Issue に照らして、甘さ・逸脱・退行の候補を出す。

**渡すもの:**

- 差分  
- Issue の受入条件  
- 関連する docs の該当箇所  

**受け取り方:**

- 指摘は **候補** であり、自動で採用しない  
- 実コードで裏取りしてから直す  

#### 5. 人間の判断（自分）

**目的:** 直してループを回すか、PR に進むかを決める。

| 判断 | 次 |
|------|-----|
| **revise** | 同じ Issue のままステップ 3（実装）へ戻る |
| **ready** | ループを抜け、PR 作成へ |

「なんとなく良さそう」で ready にしない。受入条件を一つずつ見る。

---

### Ship — 共有し、入れる

#### 6. PR を開く（Claude Code）

**目的:** レビューと履歴のために、変更を PR として提出する。

**PR に含めるもの:**

- Issue へのリンク  
- 変更の要約  
- 検証結果（テスト・ビルドなど）  
- 重要な判断（Codex 指摘を却下した理由など）  

**注意:** PR を開くのは Claude Code の仕事。**マージすることではない。**

#### 7. 人間のマージゲート（自分）

**目的:** main に入れるかどうかを決める。

| 結果 | 次 |
|------|-----|
| マージする | リポジトリごとの公開・本番手順へ進める |
| Issue へ戻す | スコープや受入が違う。ステップ 2 を直してから再開 |

ChatGPT が PR の論点整理を手伝うことはある。**マージボタンを押すのは自分。**

---

## やり直しの道（2 種類を混ぜない）

| 種類 | 意味 | 戻る先 |
|------|------|--------|
| 実装のやり直し | Issue はそのまま。差分が足りない | ステップ 3 |
| Issue のやり直し | そもそも切ったスコープが違う | ステップ 2 |

実装の不備を、黙ってスコープ変更にしてしまうと、後から再現も説明もできなくなります。

---

## よくある誤解

| 誤解 | 実際 |
|------|------|
| AI が最後まで出荷する | マージは人間 |
| 壁打ち＝仕様確定 | 壁打ちは候補。確定は Issue |
| レビュー指摘は全部直す | 採否は人間。誤指摘は却下して理由を残す |
| 巨大 PR の方が早い | 一度に一つの変更の方がレビューと差し戻しが効く |

## この文書が表す実践

private のプロダクト開発でも、「情報収集 → Issue → ブランチ → 実装 → 敵対レビュー → PR → 人間がマージ」という骨格は同じです。リポジトリごとの検証項目や本番手順は別途あります。図は、**共通する順序と責任**を示しています。
