# How I ship（日本語）

![How I ship 図](../../assets/diagrams/how-i-ship-v4.png)

一気通しの魔法ボタンではない。一度に一つの変更。Issue でスコープを切る。ツールは提案と批判をし、**出荷を決めるのは自分**。

プロファイル図の詳細説明。[English](../how-i-ship.md) · 関連: [知識が積み上がる仕組み](./how-knowledge.md) · [ここでの Claude Code](./how-claude.md)

---

## 役割（誰が何をするか）

| 役割 | ツール | 責任 |
|------|--------|------|
| **探索・批判** | ChatGPT, Codex | 壁打ち設計、Issue スコープに対する敵対レビュー |
| **実装** | Claude Code | ブランチ作業、PR 作成、PR への判断記録 |
| **制御面** | GitHub Issue | スコープ、受入条件、作業の単位 |
| **決定** | 人間（自分） | レビュー指摘の採否、マージ or Issue へ差し戻し |

図の色はこれらの役割に対応する。「AI が偉い／人間が偉い」ではない。

---

## 手順

### PLAN

1. **変更をフレーミングする（ChatGPT）**  
   問題・選択肢・粗い設計を壁打ちする。会話であり、契約ではない。

2. **スコープと受入を定義する（自分 · GitHub Issue）**  
   1 Issue に切る: 受入条件、docs へのリンク、「完了」の定義。  
   以降のすべてを縛る制御面。

### BUILD · REVIEW（ループ）

3. **実装する（Claude Code）**  
   専用ブランチでスコープ内の変更。main への直接 push はしない。

4. **敵対レビュー（Codex）**  
   Issue スコープに対して見る。指摘は候補であり命令ではない。

5. **人間の判断（自分）**  
   指摘を残すか落とすかを決める。  
   - **revise** → 実装へ戻る（同じ Issue）  
   - **ready** → ループを抜けて PR を開く

### SHIP

6. **PR を開く（Claude Code）**  
   Issue をリンクし、判断（Codex 指摘の却下理由含む）を記録する。

7. **人間のマージゲート（自分）**  
   ChatGPT がレビュー議論を手伝うことはある。**マージするのは自分**（または Issue へ差し戻し）。  
   その後、本番へ出せる。

### フィードバック

- **ループ revise:** 判断 → 実装（同じ Issue のまま）  
- **Issue revise:** マージゲート → Issue（スコープや受入の修正。世界をやり直さない）

---

## これは何か／何ではないか

- 「AI がマージしてくれる」ではない  
- 曖昧な目標の巨大 PR ではない  
- チャット履歴を正本にしない（[知識の流れ](./how-knowledge.md) 参照）

---

## 根拠・実践

private のプロダクト作業でも同じ形（Issue → ブランチ → PR → 人間マージ、PR 前の敵対レビュー）が出てくる。リポごとに細部は違う。公開図は持ち運び用の版。
