## この文書が説明すること

この文書は、**日常で拾った情報を、あとで使える判断にし、必要なときだけソフトウェア変更（Issue）につなげる**流れを説明します。

[How I ship](./how-i-ship.md) が「**一つの Issue をマージまで**」なら、このページは **その Issue が生まれる前と、結果が判断に戻る後** を扱います。

| この文書で分かること | この文書で扱わないこと |
|----------------------|------------------------|
| 何を・どこから集めるか | 個別ツールの設定手順の全網羅 |
| メモと Issue と docs の違い | すべてのリポジトリの固有ルール |
| なぜチャットを正本にしないか | 「AI が自動で知識を整理する」幻想 |

図の中の “claim” は、ここでは **証拠と用途を伴う、後から見直せる判断** を指します。

---

## なぜ「まず情報収集」か

実装や Issue から始めると、次が起きやすいです。

- 既にある実装を知らずに二重に作る  
- 制約（権限・PII・期限）を後から知って手戻りする  
- 壁打ちが一般論になり、そのリポジトリに効かない  

だから順序は:

1. **集める**（何が起きているか、何が既にあるか）  
2. **書き下ろす**（判断として残す）  
3. **必要なら共有する**（プロダクト docs / ADR）  
4. **変えるなら Issue にする**  
5. **作って確かめる**  
6. **結果で判断を更新する**

## 情報収集で何を集めるか

### 情報の種類

| 種類 | 具体例 | あとで効く場面 |
|------|--------|----------------|
| 現場・ユーザー | 問い合わせ、画面の詰まり、運用の手作業 | 何を直すかの根拠 |
| リポジトリの事実 | 既存コード、テスト、設定、docs | どこを触るか |
| 過去の判断 | ADR、古いメモ、前回の障害記録 | 同じ議論を繰り返さない |
| 外部の調査 | API の挙動、ライブラリの制約 | 実現可能性 |
| 制約 | 期限、触るな領域、セキュリティ | スコープの上限 |

### 情報の置き場所（収集の直後から分ける）

| 置き場 | 向いているもの | 向かないもの |
|--------|----------------|--------------|
| **Obsidian** | 調査メモ、仮説、個人の判断、後で見返す材料 | そのまま本番の仕様として確定した話だけ（それは repo へ） |
| **リポジトリの docs / ADR** | プロダクトに効く決定、チームが従うルール | 未整理の走り書き |
| **GitHub Issue** | 「次にコードを変える単位」と受入条件 | 調査ログの全文ダンプ |

「全部 Obsidian」「全部リポジトリ」にはしません。**寿命と共有範囲**で分けます。

## 全体の流れ

| 段階 | ステップ | 平たい意味 |
|------|----------|------------|
| **取り込む** | 1 → 2 | 気づく。あとで戻れる判断にする |
| **決める** | 3 → 4 | プロダクトに効くなら docs。コードを変えるなら Issue |
| **作る** | 5 → 6 | 実装して確かめる |
| **学ぶ** | 7 → 2 | 判断は正しかったか。メモを更新するか捨てる |

```
情報収集 → 判断を書く（Obsidian）
    → 必要なら docs/ADR
    → 変更するなら GitHub Issue
    → 実装・検証
    → 結果を見て判断を更新（昇格しないことも多い）
```

---

## 手順（図の 1〜7 を詳しく）

### 1. 証拠を観察する

**目的:** 失うと手戻りする材料を、生のまま取りこぼさない。

**実際にやることの例:**

- 再現手順やスクリーンショットを残す  
- 関係しそうなコードパスを lit する  
- 類似の過去 ADR / Issue を探す  
- 「まだ分からないこと」を明示して書く  

**終わったと言える状態:**

- 何が起きたか（または何を変えたいか）を、他人に説明できる材料がある  
- 「憶測」と「見た事実」が分かれている  

### 2. 判断を書き下ろす → Obsidian

**目的:** チャットの流れを、後から検証できる一文（または短い節）にする。

**判断に含めるもの:**

| 要素 | 例 |
|------|-----|
| 主張 | 「〜の方がよい／〜が原因らしい」 |
| 根拠 | ログ、コード参照、ユーザー報告 |
| 用途 | この判断を何に使うか（実装、優先度、調査継続） |
| リスク | 外れたときに壊れるもの |

**終わったと言える状態:**

- 一週間後の自分が、根拠まで辿れる  
- 「なんとなく」から一歩出ている  

ここが Obsidian の主戦場です。

### 3. 決定の文脈を記録する → リポジトリ（プロダクトに効くとき）

**目的:** 個人メモを超えて、**そのプロダクトの作り方が変わる決定**を残す。

**repo に上げる例:**

- 認証の方針  
- データの持ち方  
-  debounced な仕様の確定  

**まだ vault に置いたままでよい例:**

- 未検証の仮説  
- 個人用の調査メモ  

docs と Issue は混ぜません（後述）。

### 4. スコープと受入を定義する → GitHub Issue

**目的:** 「次にコードを変える単位」を、実装とレビューが共有できる形にする。

**Issue が担うこと / docs が担うこと:**

| | docs / ADR | Issue |
|--|------------|--------|
| 役割 | 何が真として採用されたか | 何を今回変えるか |
| 寿命 | 長い | その変更の間 |
| 完了 | 方針が文書として残る | 受入条件を満たして閉じる |

**終わったと言える状態:**

- [How I ship](./how-i-ship.md) のステップ 2 と同じく、受入条件が具体的  
- 実装者が Issue を正にして動ける  

### 5. ブランチで実装する

知識のページでは深く書きません。道具とゲートは [How I ship](./how-i-ship.md) を正とします。

要点だけ:

- Issue の外に出ない  
- ブランチでやる  
- Claude Code が主に実装する（詳細は [Claude Code](./how-claude.md)）  

### 6. 振る舞いを検証する

| 種類 | 例 | 誰が多いか |
|------|-----|------------|
| 自動 | tests · build | Claude / CI |
| 手元 | 画面操作、実データ、境界値 | 自分（必要なとき） |

「緑なら完了」にはしません。

### 7. 結果を評価する → 学びを Obsidian の判断へ

**目的:** 実装して分かったことで、ステップ 2 の判断を更新する。

| 結果 | 判断メモ側でやること |
|------|----------------------|
| 想定どおり | 根拠を強める、または「採用」と残す |
| 一部違う | 仮説を修正する |
| そもそも違う | 捨てる、または別の観察に戻る |

**学び → 次の判断（Obsidian）** の破線は、「全部が成功談になる」という意味ではありません。**落ちる判断の方が多い**前提です。

---

## 情報収集に戻るタイミング

次のときは、実装や Issue を進めるより、収集に戻ります。

- Issue の受入が書けない（材料不足）  
- 壁打ちが一般論しか返さない（リポジトリ事実が足りない）  
- レビューで「そもそも前提が違う」と言われる  

---

## よくある失敗

| 失敗 | 起きること | 直し方 |
|------|------------|--------|
| チャットだけ残す | 後から根拠が消える | 判断を Obsidian に書き下ろす |
| 調査ログを Issue に全部貼る | 変更の単位が読めない | Issue は受入だけ。調査は vault |
| docs と Issue を同一視する | 仕様変更と作業単位が混線する | 分ける |
| 結果を見ない | 同じ判断を繰り返す | ステップ 7 を習慣にする |

## この文書が表す実践

調査用の vault（Obsidian）、プロダクトの docs、変更の Issue を分け、実装後に判断を更新する、という骨格です。ツール名は環境で変わり得ますが、**情報収集を先に置き、チャットを正本にしない**点は変えません。
