## この文書が説明すること

この文書は、**Claude Code に何を読ませ、何をさせ、何を絶対に任せないか**を説明します。

[How I ship](./how-i-ship.md) の全体図のうち、**実装者としての Claude の境界**だけを拡大したページです。

| 分かること | 分からないこと（ここでは扱わない） |
|------------|--------------------------------------|
| Claude が読む入力 | ChatGPT 壁打ちの全手順 |
| Claude が書いてよい差分 | プロダクトロードマップの決め方 |
| PR とマージの担い手の違い | 各リポジトリ固有の CI 設定 |

---

## 前提：まず情報と Issue がある

Claude に作業を渡す前に、次が揃っていることが望ましいです。

1. **情報収集**（現状のコード・制約・きっかけ）— 詳しくは [知識の流れ](./how-knowledge.md) と [How I ship](./how-i-ship.md) の冒頭  
2. **GitHub Issue**（スコープと受入条件）  
3. 必要なら **docs / ADR**（プロダクト上の決定）  

Claude に「なんかいい感じに」とだけ渡す使い方はしません。  
**読む材料がない実装は、速いが外れる**からです。

## 役割の分割（図の上段）

### READS — 何を読むか

Claude Code が参照する主な入力:

| 入力 | 何のため |
|------|----------|
| GitHub Issue | スコープと受入の正 |
| プロジェクトルール（CLAUDE.md 等） | そのリポでの禁止・手順 |
| 関連 docs / ADR | 仕様の根拠 |
| 既存コード | 実装の足場 |

ロゴ上は GitHub（Issue / リポ）が中心です。

### DOES — 何をするか

| すること | しないこと |
|----------|------------|
| 専用ブランチでスコープ内の変更 | main への直接 push |
| 自動で回せる tests · build | スコープ外の「ついで」大改修 |
| 指摘反映のための追記コミット | 受入条件の勝手な書き換え |

**手元確認（画面操作など）**は、変更の性質によっては自分が行う。  
「tests · build が通った＝人間の確認まで終わった」とは言わない。

### DOES NOT OWN — 何を持たないか

| 持たないもの | 誰が持つか |
|--------------|------------|
| プロダクトスコープの最終決定 | 自分（Issue / docs） |
| レビュー指摘の採否 | 自分 |
| **main へのマージ** | 自分 |

破線で DOES と繋がっているのは、「次の工程」ではなく **境界** の表示です。

## 実装のあと（図の下段・How I ship と同じ順）

実装がブランチに載ったあと:

| 順 | 名前 | 担い手 | 何をするか |
|----|------|--------|------------|
| 1 | Codex review | Codex | Issue に照らして差分を突く |
| 2 | Human judgment | 自分 | 指摘を残すか捨てるか |
| 3 | Open PR | **Claude Code** | PR を開き、Issue をリンクし、経緯を書く |
| 4 | I merge | **自分だけ** | main に入れるか、Issue に戻すか |

### ここがずれやすい点

| 行為 | 担い手 | よくある誤解 |
|------|--------|--------------|
| PR を開く | Claude Code | 「人間が PR まで全部やる」 |
| マージする | 人間 | 「Claude がマージする」 |

**PR を開くことと、マージすることは別です。**

Codex の指摘は命令ではありません。実コードで裏取りし、却下するなら理由を残します（PR や Issue 側）。

## 実際の受け渡し（イメージ）

```
[情報収集・壁打ち・Issue 確定]  ← 自分 / ChatGPT / 知識の流れ
        ↓
[READS] Issue・docs・ルールを読む  ← Claude
        ↓
[DOES] ブランチで実装・tests/build  ← Claude
        ↓
[Codex] 敵対レビュー
        ↓
[自分] 指摘の採否
        ↓
[Open PR] Claude が PR を開く
        ↓
[I merge] 自分がマージ（または Issue 差し戻し）
```

---

## なぜこう分けるか

1. **速度と責任を分ける**  
   実装は速くてよい。スコープとマージの責任は人間に残す。

2. **レビューを形骸化させない**  
   敵対レビュー → 人間の採否、がないと「通ったことにする」だけになる。

3. **履歴を残す**  
   PR に Issue・判断・却下理由が残ると、後から追跡できる。

## よくある誤解

| 誤解 | 実際 |
|------|------|
| Claude がプロダクトを決めている | スコープは Issue / 自分 |
| Claude がマージしている | マージは自分 |
| 自動テストだけで完了 | 必要なとき hands-on は自分 |
| How I ship と同じ全部をこの図が表す | この図は境界の拡大図 |

## この文書が表す実践

この運用をしている製品リポジトリでは、プログラマ＝**Claude Code**（ブランチ・PR を開く）、ブレイン＝**自分**（スコープ・指摘の採否・マージ）、敵対レビュー＝**Codex**（PR 前）、という骨格です。CI や本番手順はリポジトリごとに違います。Claude がマージを所有することはありません。
